# DMBSearchViewer Architecture Decisions

## Project-specific section

When copying this file to another PageBuilder ecosystem project, update this section first.

- Project name: `DMBSearchViewer`
- Architectural role: runtime search UI and provider aggregation package.
- Main stability concerns: public routes, provider contracts, view models, result shape, provider ordering, result URL handling, navbar integration, and configuration defaults.
- Documentation target: DocumentationBuilder output rendered in `labs_idemobi_com`.

## ADR-001: Provider contracts are public extension points

- Context: Host applications and packages can add search sources through `IDMBSearchProvider`.
- Decision: Keep provider inputs, outputs, and error behavior backward compatible unless breakage is explicitly approved.
- Consequence: Provider contract changes must include documentation and migration notes.

## ADR-002: Generated SearchBuilder data is a runtime input

- Context: `DMBSearchViewer` reads search records generated by `DMBSearchBuilder`.
- Decision: Keep database assumptions aligned with `DMBSearchBuilder` and document compatibility expectations.
- Consequence: SearchBuilder database-shape changes require corresponding viewer review.

## ADR-003: Search UI must remain host-friendly

- Context: The package is embedded in host MVC applications.
- Decision: Keep routes, embedded views, navbar component behavior, and styling assumptions stable.
- Consequence: Visual or route changes require manual preview coverage and clear delivery notes.

## ADR-004: DocumentationBuilder-first documentation

- Context: Documentation is expected to be extracted and rendered into `labs_idemobi_com`.
- Decision: Public APIs must include useful XML documentation and stable references.
- Consequence: AI prepares documentation and must not claim DocumentationBuilder execution unless it was actually run.
